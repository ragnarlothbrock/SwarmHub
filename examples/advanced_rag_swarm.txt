### 1. REPLICATION TARGET ARCHITECTURE: THE "COGNITIVE BLOB" CONCEPT

A primary core tenant of this repository is the framework-agnostic "Cognitive Blob" philosophy. Rather than locking business logic, prompt formatting, or runtime conditions directly inside a single framework API (such as LangGraph's StateGraph or CrewAI's worker definitions), all cognitive logic blocks are written as pure, standard Python scripts inside the 'blobs/advanced_rag/' directory.

This allows SwarmHub to act as a cross-compiler. The developer builds a single, high-level abstract workflow spec topology ('examples/advanced_rag_swarm.py'). SwarmHub ingests that specification map and automatically emits native, production-ready implementation scripts targeting:
1. Microsoft AutoGen (Conversational Room Management Interaction Interface)
2. CrewAI (Sequential Role-Driven Automation Worker Workforce)
3. LangGraph (Functional Graph Low-Level Cyclic State Engine Topology)

### 2. HARDWARE & COMPILATION BASELINE ENVIRONMENT

All operational testing and optimization sweeps were performed entirely on-device under a zero-egress, fully offline local configuration:

### 3. DOWNSTREAM PRIMITIVE INFERENCE ENGINE SERVICE INSTALLATION

Before starting python processing, you must establish the background model serving capability dependencies via Ollama:

1. Download and install Ollama for Apple Silicon via official channels.
2. Launch the server application background daemon through your terminal interface.
3. Download the specific high-performance embedding model used by our vector daemon:
   $ ollama pull nomic-embed-text
4. Download the primary lightweight logical reasoning model driving our grader, rewriter, and final synthesis generator blocks:
   $ ollama pull llama3.2:3b

### 4. VIRTUAL ENVIRONMENT & RUNTIME DEPENDENCY SPECIFICATION

Create a localized isolated virtual Python environment (Python 3.10 or higher recommended) and install the multi-framework requirement matrix:

# Initialize and activate the isolated virtual environment shell
python3 -m venv .venv
source .venv/bin/activate

# Upgrade pipeline package managers
pip install --upgrade pip

# Install Document Extraction & Processing Dependancy Layers
pip install pypdf

# Install Core Persistent Embedded Vector Database Storage Client

# Install Microsoft AutoGen Target Execution Libraries

# Install CrewAI Native Automated Agent Workforce Infrastructure
pip install crewai

# Install LangGraph Runtime Orchestration Core & Extension Packages
pip install langgraph

### 5. EXECUTION & VERIFICATION REPRODUCTION STEPS

Follow this sequence to index your technical documentation data layers, compile the universal specifications, and verify cross-compiled target executions:

STEP 1: Populate Your Data Source Repository Vault
Drop any dense technical reference documents (e.g., standard ML papers like the original Transformer Paper 'attention_is_all_you_need.pdf' or the LoRA paper 'lora_adaptation.pdf') into your source directory:
path: examples/documents/

STEP 2: Execute the SwarmHub Master Compiling Engine Blueprint
Run the declarative composition script to map your abstract graph nodes, compile data contracts, and emit target source code files:
$ python3 examples/advanced_rag_swarm.py

This step translates your blueprint map into turn-key standalone execution files stored inside:
path: dist/advanced_rag/

STEP 3: Execute Target Runtimes with Live Interactive Prompts
Run any of your automatically cross-compiled deployment files. The script will dynamically initialize a local database checkpointer file ('dist/advanced_rag/rag_memory_vault.db'), wake up the ChromaDB MCP background sub-process pipeline, and prompt you directly for a terminal query:

To execute via the LangGraph Engine State Machine:
$ python3 dist/advanced_rag/compiled_rag_langgraph.py

To execute via the CrewAI Role-Driven Workforce:
$ python3 dist/advanced_rag/compiled_rag_crewai.py

To execute via the AutoGen Conversational Agent Chatroom:
$ python3 dist/advanced_rag/compiled_rag_autogen.py

Each engine will cleanly guide the prompt through the document_retriever, context_grader, and optional recursive query_rewriter loops based strictly on the same abstract core logic!
